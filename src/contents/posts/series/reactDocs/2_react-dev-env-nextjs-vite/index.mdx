## 들어가며

안녕하세요. 지난 1-1편에서 우리는 가장 기본적인 카운터 예제를 통해 자바스크립트와 리액트의 첫 만남을 가졌습니다. 명령형 프로그래밍과 선언형 프로그래밍의 차이를 어렴풋이 확인할 수 있었습니다.

이번 1-2편에서는 리액트 개발의 첫걸음이자 매우 중요한 단계인 '개발 환경 설정'에 대해 함께 탐구해보려 합니다. 리액트 공식문서에서는 개발 환경을 설정하는 다양한 방법을 제시하지만, 그중에서도 **Next.js**와 **Vite**라는 두 가지 대표적인 도구를 중심으로 살펴보겠습니다. 왜 이러한 선택지가 있는지, 각 방식은 어떤 장단점을 가지는지, 그리고 우리에게 어떤 환경이 더 적합할지 함께 고민하며 리액트 개발의 든든한 기반을 다져봅시다. 이 과정을 통해 리액트 생태계에 대한 이해를 넓히고, 앞으로의 학습 여정을 위한 견고한 발판을 마련할 수 있을 것입니다.

## 리액트 개발 환경 설정의 중요성

먼저 우리가 늘 사용해온 자바스크립트 개발 환경을 떠올려봅니다. HTML, CSS, JavaScript 파일을 만들고 브라우저에서 열면 끝입니다. 매우 직관적이고 간단합니다. 하지만 리액트와 같은 현대적인 프레임워크나 라이브러리 환경은 단순히 파일을 열어보는 것을 넘어섭니다. 왜 리액트는 특별한 개발 환경이 필요할까요? 잠깐 살펴보자면, 리액트는 JSX 문법을 사용하고, 모듈 시스템을 활용하며, 개발 편의를 위한 다양한 기능들을 제공합니다. 이러한 기능들을 브라우저가 바로 이해하고 실행할 수 있도록 변환(트랜스파일링)하고, 여러 파일들을 하나로 묶어주는(번들링) 과정이 필요합니다. 이러한 작업들을 자동으로 처리해주는 도구들이 바로 리액트 개발 환경을 구성하는 핵심 요소들입니다.

## 리액트 개발 환경의 주요 구성 요소

리액트 개발 환경을 구성하는 데는 몇 가지 핵심적인 도구와 개념이 있습니다.

1. **Node.js**: 자바스크립트 런타임으로, 리액트 개발에 필요한 도구들을 실행하고 패키지를 관리하는 데 사용됩니다.
2. **패키지 관리자 (Package Manager)**: 리액트 및 관련 라이브러리들을 설치하고 관리하는 도구입니다. (예: npm, yarn, pnpm)
3. **번들러 (Bundler)**: 여러 개의 자바스크립트, CSS, 이미지 파일 등을 하나의 파일로 묶어 브라우저가 효율적으로 로드할 수 있도록 합니다. (예: Webpack, Rollup)
4. **트랜스파일러 (Transpiler)**: JSX나 최신 자바스크립트 문법(ES6+)을 구형 브라우저도 이해할 수 있는 자바스크립트 문법으로 변환합니다. (예: Babel, SWC)
5. **개발 서버 (Development Server)**: 개발 중인 애플리케이션을 브라우저에서 실행하고, 코드 변경 시 자동으로 새로고침(Fast Refresh/HMR) 기능을 제공하여 개발 편의성을 높입니다.

이러한 요소들을 직접 설정하는 것은 복잡하고 시간이 많이 소요될 수 있습니다. 그래서 리액트 커뮤니티에서는 이러한 복잡성을 줄이고 개발자가 애플리케이션 로직에 집중할 수 있도록 '리액트 프레임워크'나 '빌드 도구'를 제공합니다. 오늘 우리는 그중 대표적인 두 가지, Next.js와 Vite를 살펴보겠습니다.

## Next.js로 리액트 개발 환경 구성하기

### Next.js란?

Next.js는 리액트를 기반으로 하는 **풀스택 웹 프레임워크**입니다. 단순히 리액트 앱을 만드는 것을 넘어, 서버 사이드 렌더링(SSR), 정적 사이트 생성(SSG), ISR(Incremental Static Regeneration)과 같은 다양한 렌더링 전략, API 라우트, 그리고 App Router를 통한 서버 컴포넌트(Server Components) 지원 등 강력한 기능을 내장하고 있어, 프로덕션 환경에 적합한 웹 애플리케이션을 빠르게 구축할 수 있도록 돕습니다. '프레임워크'라는 이름처럼, 리액트 개발에 필요한 거의 모든 것을 제공하여 개발자가 인프라 걱정 없이 코드 작성에만 집중할 수 있게 해줍니다.

### Next.js 개발 환경 설정하기

이제 Next.js를 사용하여 리액트 프로젝트를 만듭니다. 다음 명령어를 터미널에 입력합니다.

```bash
pnpm create next-app@latest my-next-app
```

명령어를 실행하면 몇 가지 질문을 합니다. 대부분의 경우 기본값을 선택하거나 "No"를 선택해도 괜찮습니다. 예를 들어 TypeScript 사용 여부, ESLint 설정 여부, Tailwind CSS 사용 여부 등을 물어볼 수 있습니다.

프로젝트 생성이 완료되면, 다음과 같은 메시지가 나타납니다.

```shell
Success! Created my-next-app at /path/to/my-next-app
Inside that directory, you can run several commands:

  pnpm run dev
    Starts the development server.

  pnpm run build
    Builds the app for production.

  pnpm start
    Runs the built app in production mode.

We suggest that you begin by typing:

  cd my-next-app
  pnpm run dev
```

안내에 따라 생성된 프로젝트 디렉토리로 이동하여 개발 서버를 시작합니다.

```bash
cd my-next-app
pnpm run dev
```

이제 웹 브라우저를 열고 `http://localhost:3000`으로 접속합니다. Next.js의 기본 페이지가 나타납니다.

## Next.js 방식의 특징 (리액트 개발 환경 구성)

- **1. 풀스택 프레임워크로서의 포괄적인 기능**: Next.js는 단순히 `UI` 라이브러리인 리액트를 넘어, 서버 사이드 렌더링(SSR), 정적 사이트 생성(SSG), API 라우트, 이미지 최적화, 파일 시스템 라우팅(App Router), Middleware/Edge Runtime 등 웹 애플리케이션 개발에 필요한 거의 모든 기능을 내장하고 있는 **풀스택 프레임워크**입니다. 이는 개발자가 백엔드와 프론트엔드를 통합된 방식으로 개발할 수 있게 하여 개발 워크플로우를 간소화하고, 프로덕션 수준의 복잡한 애플리케이션을 빠르게 구축하는 데 필요한 강력한 기반을 제공합니다. 개발자는 인프라 설정에 대한 고민 없이 핵심 비즈니스 로직에 집중할 수 있게 됩니다.
- **2. 유연한 렌더링 전략과 성능/SEO 최적화**: Next.js는 `SSR`, `SSG`, `ISR`, `CSR` 등 다양한 렌더링 전략을 지원하여, 애플리케이션의 특정 페이지나 컴포넌트에 가장 적합한 렌더링 방식을 선택할 수 있는 유연성을 제공합니다. 이러한 다재다능함은 **초기 로딩 성능을 개선**하고, 검색 엔진 크롤러가 웹 페이지 콘텐츠를 더 잘 파악할 수 있도록 하여 **SEO(검색 엔진 최적화)에 매우 유리**합니다. 이는 사용자 경험을 향상시키고 웹 애플리케이션의 접근성을 높이는 데 결정적인 역할을 합니다.
- **3. 자동 구성 및 개발 편의성**: Next.js는 Webpack (또는 Turbopack)과 SWC (Rust 기반) 트랜스파일러와 같은 복잡한 빌드 도구 설정을 **자동으로 처리**해 줍니다. 개발자가 번들링, 트랜스파일링, 코드 스플리팅 등 웹팩 설정을 직접 구성해야 하는 부담을 덜어주므로, "제로 설정"에 가까운 경험으로 빠르게 개발을 시작할 수 있습니다. 이러한 자동화는 개발자의 생산성을 극대화하고, 웹 개발의 진입 장벽을 낮추는 데 기여합니다.
- **4. 현대적인 라우팅 및 렌더링 전략 지원 (App Router, 서버 컴포넌트)**: 파일 시스템 기반의 App Router를 통해 서버 컴포넌트(Server Components), 레이아웃, 데이터 페칭 등 리액트 18 이후의 **현대적인 라우팅 및 렌더링 전략**을 지원합니다. 서버 컴포넌트를 사용하면 서버에서만 렌더링되는 컴포넌트를 작성하여 클라이언트 번들 크기를 줄이고, 초기 로딩 속도를 더욱 빠르게 할 수 있습니다. 이는 풀스택 리액트 개발의 새로운 패러다임을 제시하며, 고성능 웹 애플리케이션을 구축하는 데 필수적인 기능을 제공합니다.
- **5. 거대한 생태계와 활발한 커뮤니티 지원**: Next.js는 Vercel에 의해 개발 및 유지보수되며, 전 세계적으로 많은 기업과 개발자들이 사용하고 있는 검증된 프레임워크입니다. 이로 인해 **풍부한 학습 자료, 라이브러리, 그리고 활발한 커뮤니티 지원**을 받을 수 있습니다. 문제가 발생했을 때 해결책을 찾기 용이하고, 지속적인 업데이트와 개선이 이루어지므로, 장기적인 관점에서 안정적인 개발 환경을 보장합니다.

## Vite로 React 개발 환경 구성하기

### Vite란?

Vite('바이트' 아닙니다. '비트'입니다.)는 빠르고 가벼운 프론트엔드 **빌드 도구**입니다. 개발 서버 시작 속도와 Hot Module Replacement(HMR) 업데이트 속도가 매우 빠르다는 독보적인 장점을 가집니다.

이러한 빠른 속도의 핵심은 다음 두 가지 동작 원리에 있습니다.

1.  **네이티브 ESM 기반 개발 서버**: Vite의 개발 서버는 별도의 번들링 과정 없이 브라우저가 ES 모듈(ESM)을 직접 로드하게 합니다. 이는 브라우저가 실제로 요청하는 파일만을 즉시 변환하여 전달하는 **온디맨드(On-demand) 방식**으로 작동합니다. 예를 들어, 사용자가 특정 페이지에 접근하면 그 페이지와 관련된 컴포넌트들만 변환하여 로드하므로, 전체 애플리케이션을 미리 번들링할 필요가 없어 초기 기동 속도가 매우 빠릅니다.
2.  **esbuild를 활용한 의존성 프리번들링**: `node_modules`에 있는 외부 라이브러리들(예: React, React-DOM)은 esbuild를 사용해 개발 서버 시작 시 한 번만 미리 번들링해둡니다. 외부 라이브러리는 애플리케이션 코드보다 변경될 가능성이 낮으므로, 이처럼 미리 처리해두면 개발 중 불필요한 재번들링을 줄여 효율성을 높일 수 있습니다.

Next.js 또한 개발 중 온디맨드 방식과 HMR(Fast Refresh)을 제공하지만, Vite는 "네이티브 ESM 기반 개발 서버"라는 근본적인 아키텍처에서 차이를 보이며, 위 두 가지 전략의 조합으로 개발 생산성을 극대화합니다. Next.js가 풀스택 프레임워크라면, Vite는 주로 **클라이언트 사이드 렌더링(CSR) 기반의 리액트 앱을 빠르고 효율적으로 개발**하는 데 초점을 맞춘 빌드 도구이지만, 서버 측 렌더링(SSR) 역시 지원합니다.

### Vite 개발 환경 설정하기

이제 Vite를 사용하여 리액트 프로젝트를 만듭니다. 다음 명령어를 터미널에 입력합니다.

```bash
pnpm create vite@latest my-vite-app
```

이 명령어를 실행하면 먼저 프로젝트 이름을 묻고, 이어서 어떤 프레임워크를 사용할 것인지 선택하는 프롬프트가 나타납니다. 여기서는 `React`를 선택하고, 다음으로 `TypeScript` 또는 `JavaScript`와 같은 변형 옵션을 선택합니다. 이 과정을 통해 리액트 프로젝트를 초기화합니다.

프로젝트 생성이 완료되면, 다음과 같은 메시지가 나타납니다.

```shell
Done. Now run:

  cd my-vite-app
  pnpm install
  pnpm run dev
```

안내에 따라 생성된 프로젝트 디렉토리로 이동하여 필요한 패키지를 설치하고 개발 서버를 시작합니다.

```bash
cd my-vite-app
pnpm install
pnpm run dev
```

이제 웹 브라우저를 열고 `http://localhost:5173` (포트 번호는 다를 수 있습니다)으로 접속합니다. Vite의 기본 리액트 앱 페이지가 나타납니다.

### Vite 방식의 특징 (리액트 개발 환경 구성)

- **1. 초고속 개발 서버와 향상된 개발 경험**: Vite의 핵심 강점은 **ES 모듈(ESM) 기반의 온디맨드 코드 제공**과 `esbuild`를 활용한 의존성 프리번들링 덕분에 **개발 서버 시작 속도와 Fast Refresh/HMR(Hot Module Replacement) 업데이트 속도가 압도적으로 빠르다**는 점입니다. 이는 개발자가 코드 변경 후 `UI` 업데이트를 기다리는 시간을 최소화하여, 개발 과정의 흐름을 끊기지 않게 하고 생산성을 극대화합니다. 빠른 피드백 루프는 개발자가 `UI` 로직에 더 집중하고 창의적인 문제 해결에 몰두할 수 있는 환경을 조성합니다.
- **2. 가벼운 빌드 도구와 효율적인 프로덕션 빌드**: Webpack과 같이 복잡하고 무거운 번들러 대신 `Rollup`을 프로덕션 빌드 도구로 사용하여, 개발 서버의 경량성을 유지하면서도 **최적화된 번들 결과물**을 생성합니다. `Rollup`은 `Tree Shaking`(사용하지 않는 코드 제거)에 뛰어나 최종 번들 크기를 줄이고, 애플리케이션의 로딩 속도를 향상시킵니다. 이는 `SPA(Single Page Application)`에 특히 유리하며, `UI` 라이브러리의 핵심 기능에 집중하면서도 효율적인 배포를 가능하게 합니다.
- **3. 간단한 설정과 낮은 학습 곡선**: Vite는 `Next.js`와 같은 풀스택 프레임워크에 비해 **최소한의 설정으로 리액트 개발 환경을 구축**할 수 있게 합니다. 복잡한 빌드 도구 설정이나 프레임워크 고유의 아키텍처 패턴을 깊이 이해할 필요 없이, 빠르게 프로젝트를 시작하고 리액트 핵심 개념 학습에 집중할 수 있도록 돕습니다. 이는 리액트와 모던 웹 개발을 처음 접하는 개발자에게 진입 장벽을 낮춰주며, 프로토타이핑이나 소규모 프로젝트에 매우 적합한 환경을 제공합니다.
- **4. 클라이언트 사이드 렌더링(CSR) 최적화 및 `SSR` 확장성**: Vite는 `SPA` 개발에 최적화되어, 브라우저에서 `JavaScript`를 통해 `UI`를 동적으로 렌더링하는 **클라이언트 사이드 렌더링(CSR)** 방식에 뛰어난 성능을 발휘합니다. `Vite` 자체는 `Next.js`처럼 다양한 렌더링 전략을 기본적으로 제공하지는 않지만, 필요한 경우 **서버 측 렌더링(`SSR`) 기능으로 확장**할 수 있는 유연성을 가집니다. 또한 라우팅이나 데이터 페칭 같은 애플리케이션 레벨 기능은 기본 제공하지 않으므로, React Router, TanStack Query 등과 조합하는 구성이 일반적입니다.
- **5. 다양한 프론트엔드 프레임워크 지원과 유연한 생태계**: Vite는 리액트뿐만 아니라 Vue, Svelte, Lit 등 **다양한 프론트엔드 프레임워크를 기본적으로 지원**합니다. 이는 개발자가 특정 프레임워크에 종속되지 않고, 프로젝트의 요구사항이나 팀의 선호도에 따라 유연하게 기술 스택을 선택할 수 있는 환경을 제공합니다. 이러한 개방적인 생태계는 개발자에게 더 넓은 선택의 폭을 제공하고, `Vite`가 범용적인 빌드 도구로서 자리매김하는 데 기여합니다.

## Next.js vs Vite: 어떤 것을 선택해야 할까?

자, 이제 두 가지 대표적인 리액트 개발 환경 설정 방식인 Next.js와 Vite를 빠르게 확인해 보았습니다. 둘 사이에 어떤 차이가 있는지 비교해보고, 어떤 상황에서 어떤 도구를 선택하는 것이 좋을지 알아보겠습니다.

| 구분                  | Next.js (풀스택 프레임워크)                             | Vite (프론트엔드 빌드 도구)                           |
| --------------------- | ------------------------------------------------------- | ----------------------------------------------------- |
| 주요 역할             | 풀스택 웹 애플리케이션 개발 지원                        | 클라이언트 사이드 리액트 앱의 빠른 개발/빌드          |
| 렌더링 전략           | SSR, SSG, ISR, CSR 등 유연한 선택                       | 주로 CSR (SPA에 최적화), SSR 지원                     |
| 개발 서버             | 빠름 (내부 최적화된 Webpack/Turbopack)                  | 매우 빠름 (ESM 기반 온디맨드, esbuild 프리번들링)     |
| 개발 서버 아키텍처    | 번들러 기반(Webpack/Turbopack) + SWC, 증분 컴파일       | 네이티브 ESM 서버 + esbuild 의존성 프리번들링         |
| 라우팅/데이터 페칭    | 내장(App Router, API Routes/Route Handlers, fetch 캐싱) | 별도 라이브러리 조합(React Router, TanStack Query 등) |
| 트랜스파일러          | SWC (Rust 기반)                                         | esbuild (Go 기반)                                     |
| 번들러                | Webpack (또는 Turbopack)                                | Rollup                                                |
| 핵심 기능             | App Router, 서버 컴포넌트, API 라우트, 이미지 최적화    | 빠른 HMR/Fast Refresh, CSS Pre-processor 지원         |
| 학습 범위/개념 복잡성 | 리액트 외 Next.js 프레임워크 전반 이해 필요             | 리액트 핵심 및 빌드 도구 기본 개념 이해               |
| 적합한 프로젝트       | 대규모, SEO/성능 민감, 풀스택 지향 프로젝트             | 소규모~중규모 SPA, 프로토타입, 빠른 개발 선호         |

Next.js와 Vite는 각각 다른 지향점을 가지고 있습니다.

- **Next.js**는 풀스택 애플리케이션을 구축하고 싶거나, SEO, 초기 로딩 성능, API 라우트, App Router와 서버 컴포넌트 등 다양한 고급 기능이 필요한 경우에 강력한 선택지가 됩니다. 프레임워크가 제공하는 많은 기능 덕분에 개발자는 애플리케이션의 핵심 로직에 더 집중할 수 있지만, Next.js 자체의 학습 곡선이 존재할 수 있습니다.
- **Vite**는 가벼운 리액트 앱을 빠르게 시작하고 싶거나, 클라이언트 사이드 렌더링 기반의 SPA에 집중하고 싶은 경우에 매우 적합합니다. 개발 서버의 압도적인 속도와 간편한 설정은 개발 생산성을 크게 향상시킵니다. Vite는 기본적으로 CSR에 최적화되어 있지만, 서버 측 렌더링(SSR)도 지원합니다.

결론적으로, 여러분의 프로젝트 요구사항과 팀의 선호도에 따라 적절한 도구를 선택하는 것이 중요하다고 봅니다. 리액트를 처음 접하고 핵심 개념 학습에 집중하고 싶다면 Vite로 시작하는 것이 부담이 적을 수 있습니다. 반면, 바로 실전적인 웹 애플리케이션을 구축하고 싶거나 풀스택 개발에 관심이 있다면 Next.js가 좋은 선택이 될 것입니다.

## 요약

이번 1-2편에서는 리액트 개발의 첫걸음인 개발 환경 설정에 대해 탐구해 보면서, Next.js와 Vite라는 두 가지 강력한 도구를 통해 리액트 프로젝트를 생성하고 각각의 특징과 장단점을 비교 분석했습니다. 이번 편에서 다룬 개발 환경 설정의 주요 특징들을 종합하면 다음과 같습니다.

- **개발 환경의 중요성**: `JSX`, 모듈 시스템 등을 브라우저가 이해하도록 변환(`트랜스파일링`) 및 번들링 과정 필요
- **Next.js 특징**: 풀스택 프레임워크, 유연한 렌더링 전략(`SSR`, `SSG` 등), 자동 구성 및 최적화, `App Router`, 서버 컴포넌트, 방대한 생태계
- **Vite 특징**: 초고속 개발 서버, 가벼운 빌드 도구, 간단한 설정, `CSR` 최적화(`SSR` 확장 가능), 다양한 프레임워크 지원
- **선택 기준**: 프로젝트 규모, 성능/`SEO` 요구사항, 개발 팀의 선호도에 따라 `Next.js` 또는 `Vite` 선택
